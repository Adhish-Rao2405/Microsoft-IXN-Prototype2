# Prototype 2.1 Workcell State

## State Contract

The workcell state abstraction gives the planner, safety agent, UI, and narrator a shared view of the dynamic scene. It should be generated by deterministic Python code from the simulation state.

```json
{
  "task": {
    "goal": "Sort red cubes into bin A and blue cylinders into bin B",
    "status": "running"
  },
  "conveyor": {
    "running": true,
    "speed": 0.08
  },
  "pick_zone": {
    "occupied": true,
    "object_id": "obj_3"
  },
  "robot": {
    "busy": false,
    "holding": null,
    "ee_position": [0.52, 0.01, 0.81]
  },
  "bins": {
    "A": {
      "position": [0.25, 0.35, 0.75],
      "count": 1
    },
    "B": {
      "position": [0.25, -0.35, 0.75],
      "count": 0
    }
  },
  "objects": [
    {
      "id": "obj_3",
      "type": "cube",
      "color": "red",
      "position": [0.56, 0.0, 0.64],
      "on_conveyor": true,
      "target_bin": "A"
    }
  ]
}
```

## Field Semantics

### task

- `goal`: Current user-level goal, if one is active.
- `status`: One of `idle`, `running`, `blocked`, `complete`, or `failed`.

### conveyor

- `running`: Whether the conveyor is currently moving.
- `speed`: Current conveyor speed in meters per second.

### pick_zone

- `occupied`: Whether an object is currently eligible for baseline picking.
- `object_id`: The eligible object ID, or `null`.

### robot

- `busy`: Whether the robot is executing a grasp, place, or reset operation.
- `holding`: Held object ID, or `null`.
- `ee_position`: Current end-effector position.

### bins

Each bin entry contains:

- `position`: Drop pose or representative bin center.
- `count`: Number of objects successfully placed into that bin.

### objects

Each object entry contains:

- `id`: Stable object ID.
- `type`: Shape or class, such as `cube` or `cylinder`.
- `color`: Human-readable color.
- `position`: Current world position.
- `on_conveyor`: Whether the object remains conveyor-managed.
- `target_bin`: Expected bin according to sorting rules.

## State Refresh Requirements

The orchestrator should refresh state:

- Before planning.
- Before safety validation if meaningful time has elapsed.
- After every accepted action.
- After every failed action.
- Before replanning.

The UI should receive workcell state updates through the existing WebSocket path and should also be able to fetch the latest state through a REST endpoint.
